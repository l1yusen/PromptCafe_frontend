PART 03接口设计

=== slide12.xml ===
3.1 总体接口设计围绕 PromptCafe 的六个核心模块，接口设计统一采用 REST 风格、JWT 鉴权和 JSON 响应封装。统一设计原则：接口统一挂载在 /api 下，便于前后端协作与后续网关管理。成功响应采用 data 封装，失败响应采用 error.code / error.message 结构。需要登录的接口统一携带 Authorization: Bearer Token，管理员接口额外校验角色。列表接口与详情接口分离，避免一次性返回过多 Prompt 正文或敏感数据。设计重点：角色划分清晰：普通用户面向创作与分享，管理员面向审核与治理。状态字段贯穿多模块，例如 pending / approved / rejected 和 active / disabled。接口先行、数据表配合的设计方式，能更好支撑前后端并行开发。代表性接口POST  /api/auth/login      用户登录POST  /api/prompts/{id}/versions     生成版本快照POST  /api/community/share    提交社区分享申请POST  /api/admin/community/audit     管理员审核社区内容POST  /api/v1/ai/polish   AI润色Prompt

3.2 各模块接口设计—用户管理模块请求方法接口路径功能说明POST/api/auth/register用户注册POST/api/auth/login用户登录GET/api/users/me查看个人信息PUT/api/users/me编辑个人信息接口目标实现用户注册、登录、登出与身份校验等基础能力，保证系统具备完整的访问入口。为 Prompt 管理、社区功能、AI 调用等模块提供统一的用户身份认证机制。支持系统初始化管理员账号创建，为后续管理员审核与系统治理提供权限基础。设计要点通过 Authorization: Bearer Token 进行接口鉴权，未登录用户无法访问受保护资源。设置 refresh 接口用于 Token 续期，避免用户频繁重新登录。管理员初始化与普通注册流程分离，便于控制系统早期权限分配。用户管理模块主要负责用户注册、登录、身份校验，是系统安全访问和后续功能调用的基础支撑模块。

=== slide15.xml ===
3.2 各模块接口设计--Prompt 管理模块负责 Prompt 的创建、查询、编辑与删除，是整个系统最基础的业务入口。请求方法接口路径功能说明GET/api/prompts 分页/检索Prompt列表GET/api/prompts/{id} 查看单个 Prompt 详情POST/api/prompts 创建新 PromptPUT/api/prompts/{id} 更新 Prompt 内容DELETE/api/prompts/{id} 删除 PromptPOST/api/prompts/render设置 Prompt 参数值并预览渲染结果PUT/api/prompts/{id} /tags为 Prompt 添加或更新标签接口目标 支持标题、描述、system prompt、user prompt、 标签与变量定义的统一管理。 通过列表 / 详情分离减少无关字段返回，提升前端加载效率。 为版本管理、社区分享和 AI 调用提供稳定的数据来源。设计要点 列表接口仅返回摘要信息，避免在列表中直接携带完整提示词正文。 更新接口与变量、标签等结构化字段协同，便于后续扩展多类型 Prompt。

=== slide16.xml ===
3.2 各模块接口设计--Prompt 版本管理模块负责保存历史快照、比较差异并支持回滚，是保障 Prompt 可追溯性的核心模块。请求方法接口路径功能说明GET/api/prompts/{id}/versions 查看历史版本列表POST/api/prompts/{id}/versions 手动创建版本快照GET/api/prompts/{id}/versions/diff 比较两个版本差异POST/api/prompts/{id}/versions/{versionId}/rollback 回滚到历史版本DELETE/api/prompts/{id}/versions/{versionId} 删除指定版本记录接口目标 保存每次重要修改时的 Prompt 快照，支持历史追踪。 为版本对比提供结构化差异数据，便于前端展示变化字段。 确保回滚操作后仍生成新版本记录，避免覆盖式修改带来历史丢失。设计要点版本表记录 title、prompt 正文、变量、标签快照和 note 备注等关键信息。差异接口只比较指定版本，前端可聚焦显示 user prompt、variables 等核心字段。回滚并不是简单恢复旧数据，而是“读取旧版本 + 覆盖当前 + 生成新快照”的闭环流程。

=== slide17.xml ===
3.2 各模块接口设计—AI 功能模块负责用户 API Key 配置、连通性验证和 AI 调用记录，为 Prompt 优化与测试提供能力支撑。请求方法接口路径功能说明GET/api/v1/ai/config/status查询当前 AI 配置状态PUT/api/v1/ai/config/api-key保存用户 API Key 与模型配置POST/api/v1/ai/config/test测试 API Key 连通性POST/api/v1/ai/polishAI 润色 PromptPOST/api/v1/ai/testAI 测试 PromptGET/api/v1/ai/test-records查询测试历史接口目标在用户侧支持个性化模型配置，在系统侧保留默认 AI 能力兜底。对敏感配置采用加密存储与脱敏展示，避免明文泄露。记录调用行为、校验状态与使用结果，便于排查问题和统计使用情况。设计要点API Key 只在保存时进入后端，前端最多展示脱敏后的 key_mask。系统默认配置与个人配置并存，能兼顾游客体验和高级用户定制需求。调用记录表能够服务于限流、异常排查和后续成本统计。

=== slide18.xml ===
3.2 各模块接口设计—社区功能模块负责 Prompt 分享、公开展示和收藏行为，是用户内容流转到社区的重要环节。请求方法接口路径功能说明POST/api/community/share 提交社区分享申请GET/api/community/prompts 浏览社区Prompt内容POST/DELETE/api/community/prompts/{id}/favorite 收藏/取消收藏社区 PromptGET/api/community/tags/{tag}/prompts 按标签查找 PromptPOST/api/community/prompts/{communityPromptId}/reports举报社区 Prompt接口目标实现个人 Prompt 到社区内容的流转，并支持审核前快照保存。通过公开列表与收藏关系维护形成社区互动闭环。确保社区展示内容不受用户私有 Prompt 后续修改影响。设计要点community_prompts 保存标题、内容、标签等快照，保证社区版本稳定。公开列表仅查询 approved 状态的数据，审核前内容不会直接暴露给所有用户。收藏接口单独维护 user_favorites，便于后续扩展热度与推荐逻辑。

=== slide19.xml ===
3.2 各模块接口设计--管理员模块负责用户治理、社区审核、举报处理与审计日志记录，是系统运行秩序的保障模块。请求方法接口路径功能说明GET/api/admin/users 获取用户列表与状态PATCH/api/admin/users/{userId}禁用 / 恢复 用户POST/api/admin/shared-prompts/{promptId} 审核用户分享的 PromptGET/api/admin/prompts/{promptId}查看用户 Prompt 详情POST/api/admin/reports/{id}/handle 处理举报记录接口目标对用户、社区内容和举报记录形成后台管理闭环。通过状态字段和审核意见，保留每次管理动作的业务依据。通过 audit_logs 记录关键操作，保证事后可追溯。设计要点用户状态建议采用 active / disabled / deleted，兼顾封禁与逻辑删除场景。社区审核与举报处理都需要记录 handled_by / reviewed_by 和操作时间。审计日志横向覆盖用户管理、Prompt 删除、内容审核和管理员登录行为。
